Four evolution paths for the catalog skill card

Each concept keeps the same five information fields — title, teams, certificate, description, and tags — while restructuring layout, spacing, and visual weight. No workflow or interaction changes are proposed.

  • A · Sectioned Atlas — labeled zones with category accent rail
  • B · Hero Header — prominent title band, associations inline, tag tray
  • C · Badge Cloud — compact, chip-forward scanning
  • D · Split Ledger — icon rail + structured content column

Current baseline

Production .tool-card from catalog.js — icon and title in one row, then sequential chip rows (teams → certs), description, and tags. Functional but visually flat: all metadata rows compete at equal weight.

Current production card

Current hierarchy analysis

Strengths
Familiar pattern; admin actions on hover; existing chip vocabulary (triage-chip, meta-badge); category left-rail already applied via data-category-slug.
Weaknesses
Teams and certs appear as visually identical rows; description sits between two chip groups; variable card height; icon competes with title in the same flex row; tags feel disconnected at the bottom.
Scan path
Title → team chips → cert chips → description → tags (5 stops, similar visual weight on rows 2–3).

Concept A — Sectioned Atlas

Explicit horizontal zones separate identity, associations, narrative, and taxonomy. A category accent rail ties cards to their section header.

Concept A mockup

Design rationale

Hierarchy
Zone 1 (header): icon + title. Zone 2 (meta): labeled "Teams" and "Certs" groups. Zone 3 (body): description. Zone 4 (footer): tags with dashed separator.
Advantages
Fastest to parse associations; labels reduce cognitive load when scanning 18+ cards; consistent card rhythm; category rail echoes section headers.
Trade-offs
Slightly taller cards (~12px); labels add visual noise on cards with only one association type; requires new CSS zones (no JS logic change).

Concept B — Hero Header

Title and icon dominate a tinted header band. Teams and certs collapse into a single association line directly under the title. Tags live in a contained tray at the bottom.

Concept B mockup

Design rationale

Hierarchy
Title is the anchor (16px/800). Associations are secondary inline metadata. Description is the narrative middle. Tags are tertiary, boxed in a soft tray.
Advantages
Strongest title prominence; feels like a natural evolution of current cards; tag tray creates clear taxonomy boundary; hero gradient adds depth without breaking glassmorphism.
Trade-offs
Inline association line can wrap awkwardly with many teams/certs; header tint may clash with archived/orphan states; slightly less vertical compactness than C.

Concept C — Badge-Centric Compact

Minimizes chrome; pushes teams, certs, and tags into a unified chip cloud below a tight title + description block. Optimized for high-density catalog browsing.

Concept C mockup

Design rationale

Hierarchy
Title → description (read first) → chip cloud (scan second). Semantic color distinguishes team (signal), cert (meta-badge), tag (feedback) chips within one region.
Advantages
Most compact; best for 3-column grids with 50+ skills; reduces card height variance; chip-forward matches filter/toolbar vocabulary elsewhere in the app.
Trade-offs
Icon demoted to small header element; associations less structurally separated; dense chip rows can feel busy on skills with 4+ teams and 3+ tags.

Concept D — Split Ledger

Dashboard-style two-column layout: a narrow icon rail on the left, structured content on the right. Teams appear both as visual pips on the rail and as chips in the main column.

Concept D mockup

Design rationale

Hierarchy
Left rail = visual identity + team density signal. Right column = title → certs → teams (explicit) → description → tags (footer rule).
Advantages
Distinctive without breaking design system; rail creates consistent left edge alignment in grids; certs elevated above teams in content column; good for comparing skills side-by-side.
Trade-offs
56px rail reduces description width; team pips are decorative (need chips for accessibility); most structural DOM change of the four concepts.
